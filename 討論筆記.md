## 1. 背景與目標

公司為影音新聞媒體，記者從**現場拍攝片段回傳**，由新媒體編輯同仁挑選文案與選片、剪輯短影音產出。

目前痛點：新媒體編輯的工作流尚未定型，**挑文案、選片耗時**，希望導入 AI 加速。

### 核心目標
1. **縮減每日產製時間**
2. **大幅增加產片量**
3. **（硬性限制）影片品質不能下降**

### 兩大規劃方向
- **A. 影音庫管理與快速分類** — 自動化分類 + 快速搜尋片段（先做）
  - 範例情境：報導一則社會事件時，背景要放記者實地拍攝的畫面 → 系統要能用自然語言快速搜到那段特定片段。
- **B. AI 協助剪片並直接產出影片**（後做）

---

## 2. 規劃策略：先「找得到」，再「剪得快」

建議分三階段，先把**搜尋**做扎實（投報率最高、風險最低），再往自動剪輯走。

### 階段 0：資產盤點與接入（地基）
- 統一**回傳/匯入流程**（記者素材怎麼進到庫裡）。
- 統一**儲存結構**（原始檔 + 中繼資料 metadata 分離）。
- 決定**雲端 vs 地端**（牽涉敏感畫面、成本、頻寬）。

### 階段 1：AI 自動標註 + 語意搜尋（先做，本案重點）
讓每段素材進庫時，自動產生可搜尋的「描述資料」：

| 能力 | 用途 | 候選工具 |
|---|---|---|
| 語音轉文字 STT | 把記者旁白/現場聲音轉逐字稿，可全文搜 | Whisper / faster-whisper（地端免費）、Google/Azure STT（雲端） |
| 畫面理解 | 自動描述每個鏡頭「拍到什麼」（場景、物件、動作） | Gemini 影片理解（可直接吃影片）、CLIP-class 影像嵌入 |
| 鏡頭切分 | 把長素材切成可重用的短片段（shot） | ffmpeg / PySceneDetect |
| 畫面文字 OCR | 抓字幕、招牌、車牌、看板等線索 | PaddleOCR / 雲端 OCR |
| 人臉/人物（選配，需評估隱私法務） | 標記出現人物 | 需謹慎，可先不做 |
| 自動標籤 | 事件類型、地點、日期、主題分類 | LLM 依逐字稿+畫面描述自動歸類 |

**搜尋層**：
- 向量資料庫（pgvector / Qdrant / Milvus）存「逐字稿 + 畫面描述」的語意向量。
- 支援**自然語言查詢**：例如「找上週某社會事件、記者在現場拍的街景畫面」→ 回傳排序後的片段清單 + 縮圖 + 時間碼。
- 同時保留**結構化篩選**（日期、記者、地點、事件分類）。

### 階段 2：AI 輔助剪輯（後做）
- **文案 → 配畫面（B-roll matching）**：給一句旁白稿，自動建議可搭配的片段。
- **自動粗剪**：依稿子產生 rough cut 時間軸，編輯再精修（人仍在迴路裡，保品質）。
- **自動上字幕 / 摘要 / 多版本切版**（YouTube 長版、IG/Reels 直式短版）。

## 3. 參考架構（草圖）

```
記者素材回傳
   │
   ▼
[匯入/Ingest] ──► 物件儲存（原始檔）
   │
   ▼
[AI 標註管線]  ffmpeg 切鏡頭 → STT 逐字稿 → 畫面理解 → OCR → LLM 歸類
   │
   ▼
[資料層] 中繼資料 DB（結構化）＋ 向量 DB（語意）
   │
   ▼
[搜尋/瀏覽 Web UI]  自然語言搜尋＋篩選＋縮圖預覽＋一鍵帶入剪輯
   │
   ▼
[階段2] AI 輔助剪輯 → 產出
```

> 備註：公司既有技術棧偏 Next.js，搜尋 UI 可自建為 Web App。

### 工具選型 — 混合式（無 GPU、國語、低維運）
> 原則：標註走雲端 API（免顧 GPU），原始檔與資料庫留本地 NAS / 內網伺服器。

| 環節 | 雲端方案（試點先用） | 若日後轉地端的對應 |
|---|---|---|
| 語音轉文字 | 雲端 STT（Whisper API / Google / Azure，國語佳） | faster-whisper large-v3 |
| 畫面理解/描述 | **Gemini（可原生吃影片）** 或 GPT-4o vision | Qwen2.5-VL |
| 影像語意搜尋 | 雲端多模態 embedding | Chinese-CLIP |
| 鏡頭切分 | ffmpeg + PySceneDetect（本地 CPU 即可，**不必上雲**） | 同左 |
| 畫面文字 OCR | 雲端 OCR 或抽幀交給 VLM 一起做 | PaddleOCR |
| 向量 + 中繼資料 DB | **本地** Postgres + pgvector | 同左 |
| 搜尋 UI | **本地** Next.js Web App（內網） | 同左 |

> **省成本關鍵**：30GB/日不要整段上雲。先在本地用 ffmpeg 抽音訊＋切鏡頭抽關鍵影格，只把「音訊＋少量影格」送雲端分析，可大幅壓低 API 費用。

> **新增關鍵變數：雲端 API 成本**。半個月試點同時當作**費用與效果的量測基準**，再決定長期是否轉地端 GPU。

## 6. 試點計畫（最小可行版本 MVP）

**目標**：證明核心價值 —「**編輯打一句國語，就能從一批素材裡快速找到要的片段**」。
不做剪輯（階段2），只做「找得到」。

**選題原則**：挑**一個不敏感的專題**（例如生活/地方/文化類），素材量小、可上雲、無法務疑慮。

**試點流程**
1. 把該專題的原始素材放到一個資料夾（試點規模：約幾小時～半個月內可控量）。
2. 本地 ffmpeg：抽音訊 + 切鏡頭 + 抽關鍵影格（省雲端費用）。
3. 送雲端：STT 出逐字稿、VLM 描述每個鏡頭畫面。
4. 建索引：逐字稿＋畫面描述存成可搜尋資料（試點可先用輕量方案）。
5. 一個簡單搜尋頁：輸入國語 → 回傳片段清單＋縮圖＋時間碼。

**成功指標（試點要量到的數字）**
- ⏱️ 編輯「找到目標片段」耗時：AI vs 現在人工。
- 🎯 搜尋命中率：前幾筆結果有沒有對的片段。
- ✍️ 國語逐字稿/畫面描述準確度。
- 💰 這批素材的雲端 API 總費用（推估每小時素材成本）。

**試點後決策**：效果好＋費用可接受 → 擴大規模、接更多專題；費用太高或敏感顧慮大 → 評估轉地端 GPU。

## 7. 階段2：AI 輔助剪輯（編輯端）— 討論

> 前提：標註（逐字稿/畫面描述/標籤/可搜尋索引）已備妥。
> 核心原則：**AI 做組裝的苦力，人留在迴路裡守品質**。AI 不直接「render 成片」，而是交出**可編輯的時間軸**，編輯精修後出片 → 既省時又不掉品質。

### AI 輔助剪輯的成熟度階梯
| 級別 | AI 做到什麼 | 編輯還要做什麼 |
|---|---|---|
| L1 找片輔助（=階段1） | 打一句話找到片段 | 自己寫稿、自己排時間軸 |
| L2 腳本+選片配對 | 依新聞主題/報社既有文章 → 產旁白稿＋分鏡，並自動把每句話配上對應片段 | 審稿、確認選片 |
| **L3 自動粗剪（建議目標）** | 把通過的腳本＋片段 → 產出 **Premiere 可開的時間軸**（片段就定位、加標記、字幕軌） | 開 Premiere 精修：修剪、節奏、調色、真人配音 |
| L4 成片要素自動化 | 自動字幕、**多平台切版（16:9 ↔ 9:16）**、TTS 旁白、各平台標題/標籤/貼文 | 審核、微調 |
| L5 近全自動草稿 | 從一篇文章 → 直接組出各平台短影音草稿 | 把關核可（必經人工） |

### 跟 Premiere 的整合方式（三種，可疊加）
1. **產 XML/FCPXML → 匯入成序列**：AI 把片段依腳本排好、附時間碼標記與字幕，編輯在 Premiere 開啟即是粗剪。最務實的起點。
2. **Premiere 內建 AI 直接用**：文字稿剪輯（Text-Based Editing）、Auto Reframe 自動切直式、Enhance Speech 降噪——把我們的逐字稿餵進去，不必重造輪子。
3. **Premiere 外掛面板（UXP/CEP）**：編輯在 Premiere 裡輸入需求，AI 直接在當前時間軸組裝。最整合但開發量最大，後期再評估。

### 哪些動作最該交給 AI（對準兩大目標）
- **省時** → 腳本+選片配對（L2）、自動粗剪（L3）、自動字幕（L4）。
- **增量** → 一份內容自動產**多平台切版**＋各平台文案（L4），這是「不加人力卻倍增產片量」的關鍵槓桿。
- **與報社綜效** → 旁白稿可直接由既有新聞文章生成（呼應 AI 電子報專案的內容資產）。

### AI 守不住、必須留給人的部分（守品質的底線）
- 編輯判斷：哪則值得做、角度、取捨。
- 節奏與情緒：剪輯點的呼吸感、高潮鋪陳。
- 事實/法務/倫理：當事人、未成年、未證實內容。
- 配樂版權、敏感畫面把關。

### 務實的目標狀態（不是「全自動出成片」）
> 「AI 出**粗剪＋字幕＋多平台切版草稿** → 編輯精修數分鐘出片」。
> 這樣才同時達成「省時」「增量」且「品質不降」。

### 待釐清（進這階段前要知道的）
- [ ] 一支典型短影音的**結構**：純旁白＋B-roll？記者出鏡？字卡？配樂？片長？
- [ ] 旁白稿**來源**：記者自寫？編輯寫？可由報社文章生成？
- [ ] 字幕、配音是用**記者原聲**還是 TTS？
- [ ] 各平台版本目前是**人工各剪一版**，還是一版改尺寸？（決定切版自動化的省時幅度）
