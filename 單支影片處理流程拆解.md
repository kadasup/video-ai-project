## 設計總原則（先記這句）
**原始影片與重資料永遠留本地；只有「需要 AI 算力的辨識／向量化」才短暫上雲，而且上雲的是輕量衍生物（音訊、少量影格、文字），不是整支影片。**
→ 同時顧到：省錢、隱私、查詢快。

---

## 一張表看全流程

| 階段 | 動作 | 在哪做 | 一句話意義 |
|---|---|---|---|
| 0 進庫/儲存 | 原檔入 NAS、給唯一 ID、記基本資訊 | **本地** | 原檔是唯一真相，永不更動 |
| 1 預處理 | 抽音訊、切鏡頭、抽代表影格 | **本地** | 把長片拆成可重用片段、減少上雲量 |
| 2 辨識 | 音訊→逐字稿、影格→畫面描述/OCR | **雲端** | 把「說什麼/拍什麼」變成文字 |
| 3 上標籤 | 逐字稿+描述→結構化標籤 | **雲端產生** | 給「快速篩選」用的分類 |
| 4 向量化 | 文字→embedding 向量 | **雲端產生** | 讓「用意思找」變可能 |
| 5 存資料庫 | 片段中繼資料+向量入庫 | **本地** | 編輯實際查的「目錄」 |
| 6 查詢 | 一句國語→比對→回片段 | **本地為主** | 閉環：一句話找到片段 |

## 逐段說明

### 階段 0 — 進庫 / 儲存　【本地】
- **做什麼**：影片放進 NAS 固定結構（依 日期／記者／專題），產生唯一 `clip_id`，記下檔名、時長、解析度、拍攝時間、來源。
- **意義/規劃**：原檔是「唯一真相來源」，**永遠留本地、永不更動**。之後所有 AI 產出的資料，都只是「指回這個 ID＋時間碼」的描述，不會動到影片本身。
- **為何本地**：影片大、又可能敏感，沒必要上雲。

### 階段 1 — 預處理（切鏡頭）　【本地】
- **做什麼**：
  - ffmpeg 抽出音訊（16kHz）。
  - PySceneDetect 偵測鏡頭切點，把整支影片切成一個個「片段 clip」（一個鏡頭一段）。
  - 每個片段抽一張代表影格當縮圖。
- **意義/規劃**：
  - 編輯要的是**片段**，不是整支影片 → 先拆成「可重用的最小單位」。
  - 音訊／影格分離，後面才能分別丟去做語音與視覺辨識。
  - **只抽代表影格**＝大幅減少要上雲的資料量與費用（不必整片上傳）。
- **為何本地**：純檔案運算，一般 CPU 就能跑、免費，也避免整片外流。

### 階段 2 — 辨識　【雲端 / OpenAI】
分兩條同時做：
- **2a 語音辨識**：音訊 → Whisper → **逐字稿（含時間碼）**。
  - 意義：把「說了什麼」變成可搜尋文字。新聞素材的旁白、受訪內容是最關鍵的搜尋線索。
- **2b 視覺辨識**：代表影格 → GPT-4o → **畫面描述＋讀畫面文字(OCR)＋場景/物件**。
  - 意義：把「拍到什麼」變成文字（夜市空景、路口、記者出鏡…）。這正是你那個「背景想放現場實拍畫面」能被搜到的基礎。
- **為何上雲**：這兩步需要 AI 模型算力，內網無 GPU，所以送雲端。**送的是音訊＋少量影格，不是整片。**

### 階段 3 — 上標籤 / 結構化　【雲端產生 → 本地存】
- **做什麼**：把逐字稿＋畫面描述交給 LLM，產出結構化標籤：主題分類、地點、人物角色（記者/受訪者）、事件類型、關鍵字。
- **意義/規劃**：標籤負責**快速縮範圍**（像圖書館分類）；和階段 4 的語意搜尋互補——**標籤縮範圍、語意找精準**。
- **關鍵規劃**：要先定一份「**分類字典**」（有哪些主題、地點怎麼寫），AI 才會分得一致；否則同類東西被貼成不同標籤，之後篩不乾淨。

### 階段 4 — 向量化（Embedding）　【雲端產生 → 本地存】
- **做什麼**：逐字稿＋畫面描述 → `text-embedding-3` → 一串數字（向量）。
- **意義/規劃**：向量讓「**用意思找**」變可能——打「人潮擁擠的市場」也能命中描述寫「夜市滿滿都是人」的片段，不必字面相同。
- **為何需要**：關鍵字只能字面命中；向量能跨同義/近義詞，命中率高很多。這是語意搜尋的核心。

### 階段 5 — 存入資料庫　【本地】
- **做什麼**：每個片段存成一筆：`clip_id`、來源檔、起訖時間碼、縮圖、逐字稿、畫面描述、OCR、標籤、向量。存本地 DB（試點用 SQLite，之後可升 pgvector）。
- **意義/規劃**：這是編輯實際在查的「**目錄**」。
  - 重點：**DB 只存「描述與指標」，原始影片仍在 NAS**。查到後靠 `clip_id`＋時間碼回 NAS / Premiere 取片。
- **為何本地**：查詢要快、要在內網、且這些中繼資料不含完整影片，留本地最安全也省錢。

### 階段 6 — 查詢時（順帶說明）　【本地為主】
- **做什麼**：編輯打一句國語 → 這句先做一次 embedding（**這一下會呼叫雲端 API，但極小極便宜**）→ 在本地向量庫比對 → 回排序後的片段（縮圖＋來源檔＋時間碼）。
- **意義**：閉環。前面所有工，都是為了這一刻能「一句話找到片段」。

---

## 本地 vs 雲端 一眼分清

- **本地（NAS / 內網伺服器）**：原檔儲存、切鏡頭/抽音訊/抽影格、資料庫、搜尋查詢。
  → 重、私密、要快的，全留本地。
- **雲端（OpenAI）**：語音轉文字、畫面理解、產標籤、產向量。
  → 只有「需要 AI 腦力」的辨識類動作上雲，且只送輕量衍生物。

## 附錄 A — 上雲那一段，實際長怎樣（階段 2~4 展開）

### A-1 音訊 → Whisper（階段 2a）
- **送什麼**：一段音訊檔（不是整片）。
- **回什麼**：逐字稿，且**切成帶時間碼的小段**：
```
[00:00:03 - 00:00:07] 記者王小明在事故現場為您報導
[00:00:08 - 00:00:12] 這起車禍發生在今天清晨...
```
- **意義**：每一句都有時間碼 → 之後能精準對回影片的某一秒。

### A-2 一張影格 → GPT-4o（階段 2b + 3 一起做）
- **送什麼**：1 張代表影格（壓到約 512px、低細節即可）＋一段固定指令。
- **指令大概這樣寫**（要求回 JSON，方便存資料庫）：
```
你是影音新聞素材標註助手。看這張影格，回傳 JSON：
{
  "scene": "一句話描述場景",
  "action": "畫面正在發生什麼",
  "objects": ["主要人物/物件"],
  "shot_type": "遠景/中景/特寫/空景",
  "on_screen_text": "畫面上的字(字幕/招牌/車牌/時間戳)，沒有就空字串",
  "tags": ["可搜尋關鍵字"]
}
```
- **它回什麼**（範例：一張路口車禍影格）：
```json
{
  "scene": "夜間都市十字路口",
  "action": "一輛白色轎車與機車碰撞後停在路中，地面有散落物",
  "objects": ["白色轎車", "機車", "紅綠燈", "圍觀路人"],
  "shot_type": "遠景",
  "on_screen_text": "2026/03/12 04:58 路口監視器",
  "tags": ["車禍", "路口", "夜間", "交通事故", "監視器"]
}
```
- **意義**：一次呼叫同時做了「看畫面（2b）＋讀畫面文字 OCR＋給標籤（3）」三件事，省呼叫次數＝省錢。

### A-3 文字 → 向量（階段 4）
- **送什麼**：把這個片段的「逐字稿 ＋ scene ＋ action ＋ tags」串成一段文字。
- **回什麼**：一串數字（`text-embedding-3-small` 是 1536 維）。存進 DB 的向量欄位，供語意搜尋比對。

## 附錄 B — 一支影片會產生幾筆資料？

**單位是「鏡頭片段 clip」，不是「整支影片」。** 一支影片切出幾個鏡頭，就有幾筆。

> 範例：一支 5 分鐘的現場素材，假設切出 **20 個鏡頭** → **20 筆 clip 紀錄**。
> 每筆 = 1 段逐字稿 + 1 張縮圖 + 1 組 GPT-4o 描述/標籤 + 1 個向量。

API 呼叫次數（這支 5 分鐘的片）：
- Whisper：1 次（整段音訊一次轉完）。
- GPT-4o：約 20 次（每個鏡頭代表影格各一次）。
- Embedding：20 筆（可批次一次送）。

## 附錄 C — 費用體感（粗估，實際以試點帳單為準）

> 用「低細節影格 + 一個鏡頭一張」估算。價格會變動，這裡看**量級**就好。

| 項目 | 單價（約） | 一支 5 分鐘/20 鏡頭 |
|---|---|---|
| Whisper 轉稿 | 約 $0.006 / 分鐘 | ~$0.03 |
| GPT-4o 看影格 | 低細節每張僅數十~百餘 token | ~$0.03（換 **gpt-4o-mini** 可再降約 10 倍） |
| Embedding | 約 $0.02 / 百萬 token | 可忽略 |
| **合計** | | **約 $0.06~0.10／支（mini 版更低）** |

**換算每分鐘素材 ≈ 約 $0.01~0.02（Whisper 是主要成本）。**
- 以每日 30GB（約 1.5~3 小時素材）估：**每天約 $1~4、每月約數十美元量級**。
- 省錢槓桿：① 影格用低細節、② 描述改用 gpt-4o-mini、③ 一個鏡頭只送一張、④ 純靜態長鏡頭不必每秒抽。

## 附錄 D — 完全沒有文字/旁白的片段怎麼辦？（如監視器車禍、純空景）

**結論先講：這正是「畫面理解」存在的理由。** 沒有旁白的片段，Whisper 轉不出東西（逐字稿是空的），所以它能不能被搜到，**100% 靠階段 2b 的視覺描述**——而我們本來就有做這件事。上面那個車禍範例就是「零文字」也能被「車禍 路口 夜間」搜到的證明。

具體處理方式：

1. **視覺描述當主角**：逐字稿空白沒關係，向量改由 `scene + action + tags` 組成 → 照樣能語意搜尋。系統天生就handle這種狀況。

2. **動作片段要「抽多張」而非一張**：監視器車禍是「過程」，一張影格抓不住。對「無語音 + 長鏡頭/動態」的片段，改成**每 1~2 秒抽一張、或一次送 3~5 張連續影格**給 GPT-4o，請它描述「這段時間發生的經過」，把事件序列講出來。

3. **善用畫面內建資訊（OCR）**：監視器/攝影機畫面常燒錄**時間戳、地點、機台編號、車牌**——GPT-4o 同一步就讀出來當標籤（時間、地點），補足沒有旁白的線索。

4. **非語音的聲音線索（進階，選配）**：碰撞聲、警笛、尖叫雖然 Whisper 不處理，但可用「聲音事件辨識」加標籤（如「碰撞聲」「警笛」）。試點可先不做，靠視覺已足夠。

5. **檔案中繼資料**：拍攝時間、裝置、（若有）GPS → 直接當結構化標籤，不花 AI 成本。

6. **人工補一刀的後路**：極少數 AI 描述不到位的，介面上讓編輯快速補個標籤即可，不必每支都顧。

> 一句話：**有旁白就「聲音＋畫面」雙線索；沒旁白就純靠畫面，而且動態片段改抽多張看「過程」。** 純場景素材反而是這套系統最能展現價值的地方——因為人工要翻一堆無字幕的監視器畫面才是最痛的。

## 附錄 E — 重要澄清：「只送衍生物」≠ 隱私沒問題

**架構理解正確**：10 分鐘的影片**不會整支上雲**。本地先做完預處理（抽音訊、切鏡頭、抽縮圖），**只把音訊＋少量影格送上去**辨識，原始影片檔永遠留 NAS。

**但要誠實面對**：上雲的東西雖然「輕」，**內容本身仍是敏感的**——
- 送上去的**音訊 = 真實的人聲與講話內容**（誰說了什麼，原音都在）。
- 送上去的**影格 = 真實的現場照片**（人臉、車牌、當事人、傷患畫面，像素都在）。

所以雲端「看得到」的，是壓縮過、抽樣過的敏感內容，**不是看不到敏感內容**。
「只送衍生物」真正省到的是：① 資料量小、② 沒有完整連續影片外流、③ 你能精準控制「哪幾刀」上去——但**不等於去敏感化**。

### 若要真正控管隱私，要做的是這些（與上雲與否分開思考）
1. **平台條款**：OpenAI **API**（非 ChatGPT 消費版）預設**不拿你的資料做訓練**，且資料僅短期留存做濫用偵測；符合資格可申請**零資料保留(ZDR)**。（以 OpenAI 最新條款為準，可查證）
2. **上雲前先去識別**：在本地對影格**模糊人臉/車牌**再送、敏感片段降畫質——主動權在自己手上。
3. **敏感分級 → 分流**：定義「絕不外流」類別（偵查不公開、當事人、未成年、性私密等）→ 這類**改走地端模型或純人工標**，其餘才上雲。
4. **白名單/黑名單規則**：哪類專題可上雲、哪類不行，寫成明文規範，系統自動分流。
5. **最徹底＝地端 GPU**：若敏感素材占比高，長期可把辨識也搬回地端（見方案總覽路線 B），完全不外流。

> 一句話：你的**流程設計是對的**（本地預處理→只送衍生物→原片不外流）；但「隱私是否可接受」要另外靠**去識別＋敏感分級＋平台條款**來守，不是靠「沒送整片」就自動解決。這也正是 [方案總覽](方案總覽.md) 列的最大未決點。
