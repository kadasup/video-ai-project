## 一、先從幾個月開始？（建議）

**用「時長」抓量比用「月份」準**，因為次要素材只是全量的子集（月份×30GB 會高估）。

| 階段 | 範圍建議 | 目的 |
|---|---|---|
| **PoC（驗證）** | **約 50~100 小時**素材（≈ 1 個月、且限定**單一類別/路線**，如地方社會/車禍監視器） | 跑通管線、定 taxonomy、量效果與費用基準 |
| 擴大 | 約 **3 個月** | 確認規模化穩定、成本可控 |
| 全量回溯 | 視成效逐步補 | 真正活化整個舊庫 |

> 為什麼先 1 個月/單類別：把**標籤字典、品質門檻、費用/小時**先量出來，再放大才不會踩坑。先窄後寬。

**量體換算**（30GB/日 全量；次要素材通常更少）：1 月 ≈ 0.9TB、3 月 ≈ 2.7TB。原始檔**留 NAS 不搬**，主機只需暫存工作區。

## 二、需要哪些機器

> 因辨識在雲端 → **PoC 主機以 CPU 為主，不需大 GPU**。

| 角色 | 機器 | 規格建議 | 說明 |
|---|---|---|---|
| **原始素材儲存** | **NAS（沿用）** | 容量夠回溯月份；唯讀分享；網路 ≥ 2.5GbE | 原檔的唯一真相，主機唯讀掛載 |
| **PoC 處理／應用主機** ⭐ | 1 台伺服器或中高階工作站 | **CPU 12~16 核**、**RAM 64GB**（最低 32）、系統碟 1TB NVMe、**工作碟 1~2TB NVMe**、網路 2.5/10GbE | 跑 ingest、ffmpeg 預處理、呼叫 OpenAI、資料庫、搜尋頁 |
| 資料庫 | 同上主機即可 | Postgres + pgvector | PoC 不必獨立 DB 機；向量+中繼資料很小 |
| 編輯用戶端 | 現有電腦 | 瀏覽器 | 連內網搜尋頁，無特殊需求 |

**重點**
- **GPU 非必要**（辨識在雲端）；要的話只是 ffmpeg 硬體加速的入門卡，可省。
- PoC 可**先借一台現有高規 PC** 試跑，驗證後再正式採購伺服器，降低前期投入。
- OS 建議 **Ubuntu Linux**（stack 部署最順），或沿用既有 Windows。
- ⚠️ 這台 ≠ 編輯的剪輯/轉檔機（那台才需要升級顯卡 3060→5070，是另一條線）。

### 本機評估（2026-06-26 實測）→ ✅ 可用本機做 PoC
- 規格：**i7-8700（6核/12緒）、RAM 32GB、GTX 1050 Ti、Win11 Pro**。
- 結論：可行。重活在雲端，本機只跑 ffmpeg＋DB＋搜尋頁；CPU 偏舊只是批次**慢一點**（過夜跑），GPU 用不到。
- ⚠️ **C 槽僅剩 3GB**：工作資料、Postgres、縮圖、暫存**一律放 D: 或 H:**（700GB+）。
- **Linux 走 WSL2**（已具備），**不裝完整 VM**（VM 會硬切記憶體）；WSL 發行版記得移到 D:（避開快滿的 C:）。或直接原生 Windows 跑亦可。
- 切鏡頭偵測是本機最吃力步驟 → 用「降解析度只供偵測」加速。

## 三、網路架構

```
        內網（不外流原檔）
┌─────────────────────────────────────────────┐
│  [NAS 原始素材] ──(SMB/NFS, 唯讀, ≥2.5GbE)──┐  │
│                                             ▼  │
│                                  [PoC 處理／應用主機]
│                                   · ffmpeg 抽音訊/切鏡頭/抽縮圖
│                                   · Postgres + pgvector
│                                   · 搜尋 Web App (Next.js)
│   [編輯瀏覽器] ──(內網 HTTP)──────────┘        │
└───────────────────────────────┬───────────────┘
                                 │ 只送「音訊＋少量影格」
                                 ▼ HTTPS 443（白名單出向）
                          [ OpenAI API ]
```

**原則**
- **內網高速**：NAS ↔ 主機建議 2.5/10GbE（影片大、讀取量高）。
- **對外最小開放**：只允許 PoC 主機 → OpenAI 的 **HTTPS 443 出向**（IP/網域白名單），其餘封閉。
- **原檔不出網**：對外只送衍生物（音訊／影格），原始影片永遠留內網。
- 可選：API 出口走 Proxy/閘道集中記錄與管控（利於日後成本/稽核）。

## 四、軟體 Stack
- 預處理：**ffmpeg** + **PySceneDetect**（抽音訊、切鏡頭、抽關鍵影格）
- 編排：Python 批次排程（PoC 可用 DB 表當佇列，免裝 Redis）
- 雲端：OpenAI（Whisper 逐字稿、GPT-4o 看畫面/標籤、text-embedding-3 向量）
- 資料層：Postgres + pgvector（中繼資料＋向量），縮圖存本機/NAS
- 介面：Next.js 內網搜尋頁

---

## 五、建置步驟
1. 備 PoC 主機（用本機）＋取得素材：**不必先整理影音庫**——請同仁提供一批監視器/地方新聞素材丟到一個資料夾（或唯讀掛載 NAS 指定路徑）即可。建議先給一小批煙霧測試，跑通再灌一個月量。
2. 裝 stack（ffmpeg / Postgres+pgvector / Python / Next.js）＋設定 OpenAI 出向白名單。
3. **先定標籤字典（taxonomy，要設計成可擴充）**：主題/地點/事件類型固定詞表（不先定，標註會不一致）。
   - **先框一類不影響日後分類**：資料結構與管線「類別無關」，所有片段共用同一套欄位＋向量；先做一類、之後加類別只是多塞資料、不必重做。
   - 唯一要避免：prompt/標籤過度寫死單一類別 → 採「**通用結構＋每類別可加掛專屬標籤**」，類別當提示傳入、描述 prompt 保持通用。
4. 跑 **1 個月單一類別**次要素材 → 產語料庫 → 搜尋驗證。
5. 量**效果**（命中率、找片時間）與**費用**（每小時成本）→ 達標再擴大。
